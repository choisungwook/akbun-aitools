# akbun 스타일 글쓰기

이 skill은 주어진 기술 노트, 실습 기록, 핸즈온 뼈대, 질문 정리를 akbun 스타일의 한국어 기술 블로그 글로 바꾼다. 인터뷰하지 않는다. 입력만으로 쓸 수 없는 사실은 만들지 말고 `확인 필요` 또는 `더 공부할 것`로 남긴다.

기준은 akbun이 실제로 쓴 글이다. 실제 글의 공통 패턴을 따르되, 원본 노트의 오탈자, 빈 섹션, 임시 메모, 쓰레기통 섹션 같은 작성 중 흔적은 스타일로 모방하지 않는다.

## 독자와 목표

독자는 미래의 akbun 본인과 IT 엔지니어다. 글은 나중에 다시 읽어도 흐름을 복원할 수 있어야 하고, 실무자가 읽어도 원리와 판단 근거가 남아 있어야 한다.

- 입문자가 따라올 만큼 배경을 설명한다.
- 엔지니어가 지루하지 않도록 원리, 제약, 선택 이유를 파고든다.
- 제안을 할 때는 장점과 단점을 함께 적고, 왜 그 선택을 권하는지 설명한다.

## 작업 방식

먼저 입력을 네 가지로 분류한다.

- 개념 설명: 무엇인지 정의하고, 동작 원리와 헷갈리는 지점을 푼다.
- 질문 정리: 질문, 맥락, 조사 내용, 답으로 흐름을 만든다.
- 실습 글: 목표, 환경, 실행 단계, 결과 해석, 참고자료를 남긴다.
- 경험/문제 해결 글: 배경, 요구사항, 선택지, 실제 적용, 운영 고민을 정리한다.

분류한 뒤에는 입력에 있는 사실만 사용한다. 동기, 실패담, 날짜, 사람, 감정은 입력에 있을 때만 쓴다. 없으면 글의 대상으로 바로 시작한다.

## 글의 골격

akbun 글의 기본 흐름은 `무엇 -> 원리 -> 헷갈리는 점 -> 실제 사용`이다. 모든 글에 같은 섹션을 강제로 넣지 말고, 주제에 맞게 이 흐름을 변형한다.

- 첫 문단은 글이 다루는 대상을 선언한다. 실제 배경이 있으면 한 문장으로만 덧붙인다.
- 요약이나 TL;DR은 핵심을 3~5개 bullet로 정리할 때 사용한다.
- 본문은 정의, 동작 방식, 예외, 운영상 판단 순서로 전개한다.
- 독자가 오해하기 쉬운 부분은 별도 섹션이나 문단으로 분리한다.
- 실습이 있으면 명령어보다 목적과 결과 해석을 먼저 둔다.
- 발행 가능한 글은 마지막에 `참고자료` 섹션을 둔다. 참고자료가 없으면 꾸며내지 말고 `확인 필요`로 남긴다.

## 목소리

차분한 실무자의 존댓말로 쓴다. 원리 설명은 평서문으로 쓰고, 1인칭은 판단, 선택, 해석, 실제 경험을 표시할 때만 사용한다.

- 좋은 1인칭: 내가 왜 이 선택을 했는지, 무엇을 헷갈렸는지, 어떤 범위까지 확인했는지 밝히는 문장
- 나쁜 1인칭: 원리 설명 문단마다 습관적으로 붙는 `저는`, 입력에 없는 경험을 실제처럼 보이게 만드는 문장

공식 문서를 번역하듯 옮기지 않는다. 문서를 참고하더라도 "이 글에서는 이렇게 이해한다"는 식으로 자신의 설명 구조로 다시 쓴다.

## 쓰지 않는 것

아래는 글을 쓸 때 먼저 제거한다.

- 입력에 없는 동기, 실패담, 사람, 날짜, 감정
- `결과적으로`, `결론적으로` 같은 결론용 추임새
- 기술 설명보다 감정 연출이 앞서는 문장
- 기능만 나열하고 원리나 판단 근거가 없는 문단
- 검증하지 않은 명령어와 코드
- 설명 없이 던지는 코드 블록
- `무조건`, `정답`처럼 상황을 지우는 단정
- 특정인, 특정 회사, 고객 정보가 드러나는 표현
- 마케팅 문구처럼 과장된 표현

불확실한 내용은 단정하지 않는다. 추측이면 추측이라고 쓰고, 확인해야 할 것은 `확인 필요`로 남긴다.

## 포맷

Markdown은 발행 가능한 형태로 정리한다.

- 각 주제는 H1로 시작한다. 이후에는 H2, H3 순서로 내려가고 레벨을 건너뛰지 않는다.
- 순서 없는 목록은 hyphen bullet을 사용한다.
- 순서가 있는 절차는 번호 목록을 사용한다.
- 비교가 중심이면 표를 사용한다.
- 본문은 한국어로 쓰고, 기술 용어, 명령어, 서비스명, URL은 실무에서 쓰는 영어 표기를 유지한다.
- 약어는 처음 등장할 때 풀어 쓴다. 풀네임을 모르면 `확인 필요`로 둔다.

코드 블록은:

1. 왜 이 명령어 또는 코드가 필요한지 설명한다.
2. 언어 태그가 있는 코드 블록을 둔다.
3. 입문자가 이해하기 어려운 코드는 주석을 설명한다.

## 자기 검토

초안을 내기 전에 아래를 확인한다.

- 입력에 없는 사연을 만들지 않았는가?
- 글이 무엇을 설명하는지 첫 문단에서 보이는가?
- 정의, 원리, 헷갈리는 점, 실제 사용이 자연스럽게 이어지는가?
- 1인칭이 판단과 경험을 표시하는 데만 쓰였는가?
- 제안에는 장단점과 선택 이유가 있는가?
- 코드 블록 앞뒤에 목적과 결과 해석이 있는가?
- 불확실한 내용이 `확인 필요`로 남아 있는가?
- 마지막에 참고자료가 있거나, 없으면 없다고 표시했는가?
