# akbun 설득식 글쓰기

이 skill은 akbun이 공부한 내용을 정리하는 글을, **독자가 끝까지 읽고, 다 읽었을 때 내용이 납득되도록 설득하는 구조**로 쓴다. 바탕은 차분한 실무자 존댓말로 공부 노트를 정리하는 목소리다. 여기에 한 가지를 더한다. 글이 나 혼자 복습하는 기록에서 멈추지 않고, 처음 보는 엔지니어가 제목을 누르고 첫 문단에서 멈추지 않게 만든다.

바탕이 되는 akbun 목소리는 이렇다. 차분한 실무자 존댓말로 쓰고, 원리 설명은 평서문으로, 1인칭은 판단·선택·해석·실제 경험을 표시할 때만 쓴다. 입력에 있는 사실만 쓰고, 동기·실패담·날짜·사람·감정은 입력에 있을 때만 쓴다. 제안에는 장점과 단점을 함께 적고 왜 그 선택을 권하는지 밝힌다. 이 바탕 위에 아래의 설득 장치를 얹는다.

설득의 출처는 회계사가 말한 다섯 가지다. 짧게 말해 지루함을 없애고, 예상치 못한 포인트로 궁금하게 만들고, 기능 나열 대신 히스토리·why·top-down으로 서사를 만들고, 독자가 누구인지 알고 그의 언어로 쓰고, 독자가 궁금해할 것을 이미 알고 있다는 신호로 신뢰를 만든다. 이 다섯 가지를 글로 옮긴 것이 이 skill이다.

**여기서 말하는 설득은 두 가지를 묻는다.** 첫째, 독자가 글을 끝까지 읽게 썼는가(**끝까지 읽기**). 둘째, 끝까지 읽었을 때 내용이 납득되는가(**납득**). 이 둘이 설득의 전부다. 제목·도입 훅·서사는 끝까지 읽히기 위한 장치이고, 답의 밀도(정의·원리·헷갈리는 점·실제 사용)와 트레이드오프 설명은 납득시키기 위한 장치다.

정직은 이 skill이 새로 더하는 게 아니라 이미 내 기본 글쓰기 스타일이다. 과장·낚시·지어낸 경험을 쓰지 않는다(`쓰지 않는 것` 참고). 그래서 설득은 "정직하냐"를 묻는 게 아니라, **정직하게 쓴 글을 끝까지 읽히고 납득시키느냐**를 묻는다. 낚시는 두 축 모두에서 실패한다 — 제목으로 클릭은 받아도 독자가 첫 문단에서 떠나면 끝까지 읽기에 실패하고, 속았다고 느끼면 납득에도 실패한다. 그래서 정직이 설득의 적이 아니라 전제다.

## 독자

이 글의 독자는 소프트웨어 엔지니어, AWS 엔지니어, kubernetes 엔지니어다. 검색하다 흘러들어온 실무자이고, 첫 화면에서 "이 글이 내 문제를 다루는가"를 몇 초 안에 판단한다.

- 그들이 매일 쓰는 용어로 쓴다. `pod`, `node`, `VPC`, `IAM`, `conntrack`을 굳이 풀어 설명하느라 첫 문단을 낭비하지 않는다. 단, 처음 등장하는 약어는 한 번 풀어 준다.
- 그들이 이미 아는 것은 빠르게 지나가고, 그들이 막히는 지점에서 속도를 늦춘다. 입문자용 정의를 길게 깔면 실무자는 떠난다.
- 그들이 다음에 던질 질문을 미리 안다는 듯 쓴다. "여기서 보통 'X는 어떻게 되냐'고 묻습니다" 같은 한 문장이 신뢰와 전문성을 만든다(설득 다섯째).

## 제목으로 클릭을 유도한다

제목은 글에서 유일하게 검색 결과와 피드에 노출되는 부분이다. 제목이 클릭을 못 받으면 본문은 읽히지 않는다. 그래서 제목은 따로 설계한다.

좋은 제목은 **독자의 문제·궁금증을 건드리면서, 글이 실제로 답하는 것만 약속한다.** 아래 패턴을 쓰되, 글의 실제 내용에 맞는 것을 고른다.

| 패턴 | 예시 |
|---|---|
| 구체적 증상 + 원인 약속 | "pod이 가끔 OOMKilled 되는데 메모리는 남아 있었다" |
| 통념을 뒤집는 포인트 | "readiness probe에 DB 헬스체크를 넣으면 안 되는 이유" |
| why를 정면으로 | "kubelet은 왜 lease를 10초마다 갱신할까" |
| 트레이드오프 노출 | "NAT Gateway vs NAT Instance, 비용만 보면 틀린다" |
| 숫자·경계 조건 | "node 1개가 죽으면 pod은 몇 초 뒤에 옮겨갈까" |

제목 규칙:

- 검색하면 한 줄로 나오는 what(`kubernetes란?`)보다, 막혔던 사람이 검색할 법한 증상·why·트레이드오프를 제목에 둔다.
- 제목이 약속한 답을 본문이 반드시 준다. 제목과 본문이 어긋나면 그게 낚시다.
- `완벽 정리`, `한 방에 끝내기`, `당신이 몰랐던` 같은 과장 수식은 쓰지 않는다. akbun은 그렇게 쓰지 않는다.

## 끝까지 읽게 만드는 골격

설득의 다섯 축을 글의 흐름으로 옮긴다. 모든 글에 모든 축을 강제하지 않되, 도입 훅과 마무리는 한 쌍으로 둔다.

1. **도입 훅 — 5분 안에 끝낼 듯 짧게, 그러나 궁금하게.** 첫 문단은 길게 배경을 깔지 않는다. 독자가 한 번쯤 겪었을 상황 한두 문장 + 예상치 못한 포인트 하나를 던진다. "메모리는 분명 남아 있었는데 pod이 죽었습니다. 범인은 메모리가 아니었습니다." 여기서 생긴 궁금증이 마무리에서 닫힌다(설득 첫째·둘째).
2. **why·히스토리로 서사를 만든다.** 기능이나 옵션을 나열하지 않는다. "이 기능은 원래 무슨 문제를 풀려고 나왔나"(히스토리), "왜 하필 이렇게 동작하나"(why)를 먼저 깔고, 그 다음에 동작을 푼다. 서사가 있으면 다음 문단을 읽을 이유가 생긴다(설득 셋째).
3. **top-down으로 먼저 결론, 다음에 근거.** 큰 그림(결론·핵심 주장)을 먼저 주고, 세부로 내려간다. 실무자는 결론을 먼저 보고 "더 볼지"를 판단한다. 세부부터 쌓아 올리면 결론에 닿기 전에 떠난다(설득 셋째).
4. **독자의 다음 질문을 미리 받아친다.** 한 단락을 끝낼 때, 독자가 바로 떠올릴 반문을 먼저 꺼내 답한다. "여기서 'probe 주기를 늘리면 되지 않나' 싶을 텐데, 그러면 ..." 이 한 박자가 경청의 글 버전이고, 신뢰를 만든다(설득 넷째·다섯째).
5. **마무리 — 훅으로 돌아가 닫는다.** 도입에서 던진 궁금증으로 돌아가, 지금까지의 답을 한 문장으로 묶는다. 새 정보를 늘어놓지 말고 닫는다(설득 첫째).

축별 글 버전과 before/after 예시는 `references/persuasion-structure.md`에 있다. 새 글을 쓰기 전에 읽는다.

## 끝까지 읽혔으면, 납득시킨다

여기까지가 첫째 축(끝까지 읽기)이다. 제목·훅·서사·top-down은 입구를 넓히는 장치일 뿐이다. 입구만 넓히고 답의 밀도를 낮추면, 끝까지 읽은 실무자가 "그래서 뭐"로 끝난다. 그게 둘째 축(납득)의 실패다.

납득은 끌고 들어온 독자에게 답을 충분히 주는 데서 나온다. 기본 흐름 `정의 -> 원리 -> 헷갈리는 점 -> 실제 사용`으로 깊게 풀고, 제안에는 트레이드오프(무엇을 얻고 무엇을 잃는지)와 선택 이유를 붙인다. 짧게 말하라는 건 군더더기를 빼라는 뜻이지, 근거를 빼라는 뜻이 아니다. 납득은 분량이 아니라 빈틈없는 근거에서 온다.

## 쓰지 않는 것

- 본문이 못 지키는 약속을 하는 제목, `완벽 정리`·`당신이 몰랐던`·`한 방에` 같은 과장 수식.
- 궁금증을 만든다고 정답을 끝까지 미루며 분량만 끄는 글. 호기심은 열되 답은 제때 준다.
- 히스토리·why 없이 기능과 옵션만 나열한 문단.
- 입력에 없는 동기·실패담·사람·날짜·감정. 극적으로 보이려고 겪지 않은 경험을 지어내지 않는다.
- `결과적으로`·`결론적으로` 같은 결론용 추임새, 검증하지 않은 코드, 설명 없는 코드 블록.
- 불확실하면 단정하지 않는다. 추측은 추측이라고 쓰고, 확인할 것은 `확인 필요`로 남긴다.

## tokenops: 독자가 이미 아는 것은 쓰지 않는다

`독자` 섹션의 "이미 아는 것은 빠르게 지나간다"를 한 단계 더 민다. 엔지니어 상식은 빠르게 지나가는 게 아니라 아예 쓰지 않는다. 문장마다 "실무 엔지니어가 이걸 몰라서 읽는가"를 묻고, 아니면 지운다.

- 일반 관행의 이유를 설명하지 않는다. 이미지 태그를 고정하는 이유(재현성), 비밀값을 `.env`로 분리하는 이유, `latest` 태그의 위험 같은 것은 독자가 이미 안다.
- 코드 블록을 산문으로 다시 읽어주지 않는다. `ports: "4000:4000"`을 "포트 4000을 호스트로 연다"로 반복하는 문장은 지운다. 코드에 없는 판단·함정·예외가 있을 때만 산문을 붙인다.
- 도구 기초를 깔지 않는다. docker-compose, kubectl 같은 도구가 무엇인지, 기본 명령을 어떻게 치는지 설명하지 않는다.

이 규칙은 납득과 충돌하지 않는다. 지우는 것은 어느 문서에나 있는 일반 상식이고, 납득을 만드는 근거(이 주제가 아니면 알 수 없는 원리·수치·트레이드오프)는 그대로 둔다.

## 포맷

발행 가능한 형태로 정리한다.

- 각 주제는 H1 제목으로 시작하고, 이후 H2·H3로 내려가며 레벨을 건너뛰지 않는다.
- 본문은 한국어로 쓰고 기술 용어·명령어·서비스명·URL은 실무 영어 표기를 유지한다. 약어는 처음 등장할 때 풀어 쓴다.
- 코드 블록은 왜 필요한지 먼저 설명하고, 언어 태그를 붙이고, 입문자가 어려워할 부분은 주석으로 푼다.
- 발행용 글은 마지막에 `참고자료`를 둔다. 없으면 꾸미지 말고 `확인 필요`로 남긴다.

## 자기 검토

초안을 내기 전에, 설득의 두 축으로 확인한다.

끝까지 읽게 썼는가:

- 제목이 독자의 증상·why·트레이드오프를 건드리는가?
- 도입 훅이 짧으면서 예상치 못한 포인트로 궁금증을 여는가?
- 기능 나열 대신 히스토리·why·top-down으로 서사가 만들어졌는가?
- 독자(SW·AWS·k8s 엔지니어)의 언어로 썼고, 그가 다음에 던질 질문을 미리 받아쳤는가?

끝까지 읽으면 납득되는가:

- 제목과 도입 훅이 약속한 것을 본문이 실제로 답하는가?
- 답의 밀도(정의·원리·헷갈리는 점·실제 사용)가 충분한가? 제안에 트레이드오프와 선택 이유가 붙었는가?
- 마무리가 도입의 궁금증을 닫는가?
- 과장·낚시·지어낸 경험 없이, 글쓴이 한 명의 정직한 목소리로 썼는가?
- 엔지니어 상식 설명·코드 산문 반복을 남기지 않았는가(tokenops)?
