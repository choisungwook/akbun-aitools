# 설득식 글 골격 (annotated)

설득은 두 가지를 묻는다. 독자가 **끝까지 읽게** 썼는가, 끝까지 읽으면 내용이 **납득**되는가. 아래 골격에서 0~3은 끝까지 읽히기 위한 장치이고, 각 섹션의 답 밀도(정의·원리·헷갈리는 점·실제 사용)와 트레이드오프 설명이 납득을 만든다.

이 골격은 누군가의 설득 5원칙을 글로 옮긴 것이다. `{...}`는 공부한 내용으로 채우는 자리다. 화자는 글쓴이 한 명이고, 겪지 않은 경험이나 가상의 인물을 만들지 않는다.

설득 5원칙과 글의 대응은 다음과 같다.

| 누군가의 설득 (말하기) | 이 skill의 글쓰기 |
|---|---|
| 짧게 말해 지루함을 없앤다 | 도입을 짧게, 마무리에서 새 정보 늘리지 않고 닫기 |
| 예상치 못한 포인트로 궁금하게 | 도입 훅에 통념을 뒤집는 한 줄 |
| 히스토리·why·top-down 서사 | 기능 나열 대신 "왜 나왔나 → 왜 이렇게 동작하나 → 결론 먼저" |
| 좋은 질문·경청으로 상대 파악 | 독자(SW·AWS·k8s)의 언어로 쓰고, 그의 다음 질문을 미리 받아치기 |
| 궁금해할 걸 이미 안다는 신호 | "여기서 보통 X를 묻습니다" 한 문장으로 신뢰 |

## 0. 제목 (필수, 따로 설계)

제목은 검색 결과·피드에서 유일하게 노출되는 부분이다. 독자의 증상·why·트레이드오프를 건드리되, 본문이 지킬 약속만 한다.

```text
{막혔던 사람이 검색할 법한 증상, 또는 통념을 뒤집는 포인트, 또는 정면 why}
```

나쁜 예 → 좋은 예:

- `쿠버네티스 probe 정리` → `readiness probe에 DB 헬스체크를 넣으면 안 되는 이유`
- `NAT 비교` → `NAT Gateway vs NAT Instance, 비용만 보면 틀린다`
- `OOM 완벽 정리` → `메모리는 남아 있었는데 pod이 OOMKilled 됐다`

## 1. 도입 훅 (필수) — 짧게 + 예상치 못한 포인트

배경을 길게 깔지 않는다. 독자가 겪었을 상황 한두 문장 + 통념을 뒤집는 한 줄을 던지고, 글 전체를 관통하는 궁금증을 연다.

```text
{독자가 한 번쯤 겪었을 상황 한두 문장}. {그런데 통념과 어긋나는 의외의 사실 한 줄}. {그 궁금증을 한 문장으로}.
```

이 궁금증이 마무리에서 닫힌다. 훅과 마무리는 한 쌍이다. 낚시로 비틀지 않는다.

## 2. why·히스토리로 서사 만들기 (필수, 반복)

기능·옵션을 나열하지 않는다. 묶음 하나를 섹션 하나로 만들되, 서사 순서로 푼다.

```text
## {주제}

(히스토리) {이 기능·동작은 원래 무슨 문제를 풀려고 나왔나}.
(why) {그래서 왜 하필 이렇게 동작하나}.
(top-down) {결론·핵심 주장을 먼저}. **{핵심 한 문장은 굵게}.**
(근거) {그 다음 정의 → 원리 → 헷갈리는 점 → 실제 사용으로 깊게}.
```

규칙:

- 큰 그림(결론)을 먼저 주고 세부로 내려간다. 세부부터 쌓으면 결론 전에 독자가 떠난다.
- 서사로 입구를 넓힌 뒤 답의 밀도를 낮추지 않는다. 훅은 입구일 뿐이다.

## 3. 독자의 다음 질문 받아치기 (권장, 반복)

한 단락을 닫을 때 독자가 바로 떠올릴 반문을 먼저 꺼내 답한다. 경청의 글 버전이고, 전문성 신호다.

```text
{여기서 보통 "{독자의 반문}" 하고 묻습니다}. {그런데 그러면 ... 하고 바로 답}.
```

## 4. 마무리 종합 (필수) — 훅으로 돌아가 닫기

도입에서 던진 궁금증으로 돌아가, 지금까지의 답을 한 문장으로 묶는다. 새 정보를 늘어놓지 말고 닫는다.

```text
정리하면, {훅의 궁금증에 대한 답}. {그래서 실무에서 어떻게 보면 되는지 한 줄}.
```

## 5. 참고자료 (발행용일 때 필수)

발행 가능한 글은 마지막에 `참고자료`를 둔다. 없으면 꾸미지 말고 `확인 필요`로 남긴다.

## 섹션 순서 요약

| 순서 | 섹션 | 필수 |
|---|---|---|
| 0 | 제목 (증상·why·트레이드오프, 약속만) | 필수 |
| 1 | 도입 훅 (짧게 + 예상치 못한 포인트) | 필수 |
| 2 | why·히스토리·top-down 서사 (반복) | 필수 |
| 3 | 독자의 다음 질문 받아치기 (반복) | 권장 |
| 4 | 마무리 종합 (훅에 답) | 필수 |
| 5 | 참고자료 | 발행용일 때 필수 |

## 짧은 예시

```text
# 메모리는 남아 있었는데 pod이 OOMKilled 됐다

container 메모리 사용량은 limit의 절반이었습니다. 그런데도 pod은 OOMKilled로 죽었습니다. 범인은 메모리 총량이 아니었습니다. 그럼 무엇이었을까요?

## cgroup은 "총량"이 아니라 "그 순간"을 본다

(히스토리) OOM killer는 원래 호스트 전체가 메모리를 다 쓰기 전에 프로세스를 정리하려고 나왔습니다. (why) cgroup v2의 memory.max는 평균이 아니라 그 순간의 사용량을 봅니다. (top-down) 즉 평균이 낮아도 짧은 순간 limit을 넘으면 죽습니다. **OOMKilled는 평균이 아니라 피크의 문제입니다.**

여기서 보통 "그럼 limit을 늘리면 되지 않나" 하고 묻습니다. 그런데 그러면 node 단위 over-commit이 커져서, 정작 다른 pod이 evict될 위험이 생깁니다. ...

정리하면, OOMKilled는 메모리가 남았는지가 아니라 그 순간 limit을 넘었는지의 문제입니다. 그래서 평균 그래프 대신 피크와 request/limit 간격을 봐야 합니다.

# 참고자료
- {확인 필요}
```
