# akbun 질문식 글쓰기

이 skill은 akbun이 공부한 내용을 정리하는 글을, **질문이 글을 끌고 가는 구조**로 쓴다. 차분한 실무자 존댓말로 공부 노트를 정리하는 글에, 한 가지를 더한다. 독자가 한 번쯤 품었을 궁금증을 먼저 꺼내고, 공부한 내용으로 그 질문에 답하며 진행한다.

질문식으로 쓰는 이유는 두 가지다. 궁금증을 먼저 건드리면 읽는 사람이 답을 찾으러 끝까지 따라온다. 그리고 질문 -> 답 리듬은 akbun 글의 `무엇 -> 원리 -> 헷갈리는 점 -> 실제 사용` 흐름과 자연스럽게 맞물린다.

인터뷰나 대담이 아니다. 화자는 글쓴이 한 명이고, 여러 사람의 발언을 지어내지 않는다. 질문은 글쓴이가 독자(또는 과거의 자신)를 대신해 던지는 것이지, 가상의 인터뷰이가 말하는 게 아니다.

## 독자와 목소리

독자는 미래의 akbun 본인과 IT 엔지니어다. 글은 나중에 다시 읽어도 흐름을 복원할 수 있어야 하고, 실무자가 읽어도 원리와 판단 근거가 남아 있어야 한다.

- 입문자가 따라올 만큼 배경을 설명하고, 엔지니어가 지루하지 않도록 원리·제약·선택 이유를 파고든다.
- 차분한 실무자 존댓말로 쓴다. 원리 설명은 평서문으로, 1인칭은 판단·선택·해석·실제 경험을 표시할 때만 쓴다.
- 제안에는 장점과 단점을 함께 적고, 왜 그 선택을 권하는지 밝힌다.

포맷도 발행 가능한 형태로 정리한다.

- 각 주제는 H1 제목으로 시작하고, 이후 H2·H3로 내려가며 레벨을 건너뛰지 않는다.
- 본문은 한국어로 쓰고 기술 용어·명령어·서비스명·URL은 실무 영어 표기를 유지한다. 약어는 처음 등장할 때 풀어 쓰고, 모르면 `확인 필요`로 둔다.
- 코드 블록은 왜 필요한지 먼저 설명하고, 언어 태그를 붙이고, 입문자가 어려워할 부분은 주석으로 푼다.
- 발행용 글은 마지막에 `참고자료`를 둔다. 없으면 꾸미지 말고 `확인 필요`로 남긴다.

## 질문이 글을 끌고 가는 골격

네 부분으로 전개한다. 모든 글에 모든 부분을 강제하지 않되, 도입 훅과 마무리 종합은 한 쌍으로 둔다.

1. 도입 훅 — 독자가 한 번쯤 겪었거나 들어봤을 상황을 한두 문장으로 깔고, 글 전체를 관통하는 질문 하나를 던진다. `왜 그럴까요?`, `정말 그럴까요?`처럼 궁금증을 여는 질문이다. 이 질문이 마무리에서 답으로 닫힌다.
2. 질문으로 여는 섹션 — 본문은 소제목을 질문으로 두거나, 문단 첫 문장을 질문으로 연 뒤 공부한 내용으로 답한다. 질문이 다음 내용을 끌어온다.
3. 답의 밀도 — 질문은 입구일 뿐이다. 답은 정의 -> 원리 -> 헷갈리는 점 -> 실제 사용으로 푼다. 질문을 붙였다고 답의 밀도를 낮추지 않는다.
4. 마무리 종합 — 도입 훅의 질문으로 돌아가, 지금까지의 답을 묶어 결론을 낸다.

annotated 골격과 예시는 `references/question-flow.md`에 있다. 새 글을 쓰기 전에 읽는다.

## 질문 설계

질문은 검색하면 바로 나오는 what이 아니라, 답을 찾는 과정에서 배우는 why·what-if로 만든다. what 질문은 답이 한 줄이라 글을 끌지 못한다.

| 유형 | 글을 끄는 질문 예시 |
|---|---|
| 설계 의도 | "왜 kubelet은 lease를 10초마다 갱신할까?" |
| 트레이드오프 | "readiness probe에 DB 상태를 넣으면 무엇을 얻고 무엇을 잃을까?" |
| 경계 조건 | "모든 node가 동시에 not ready가 되면 pod은 어디로 갈까?" |
| 비교 | "Gateway API는 Ingress와 무엇이 다르고 왜 나왔을까?" |
| 연결 | "eBPF로 kube-proxy를 대체하면 conntrack에는 어떤 영향이 있을까?" |

한 섹션은 질문 하나로 좁힌다. 질문이 넓으면(예: "쿠버네티스 네트워킹은 어떻게 되는가?") 답이 흩어진다. 좁은 질문이 답을 선명하게 만든다.

## 질문을 쓰는 형태

- 소제목 질문: 섹션 헤더 자체를 질문으로 둔다. `그건 의지의 문제일까?`처럼 단정 대신 궁금증으로 연다.
- 문단 여는 질문: 문단 첫 문장을 질문으로 던지고 바로 답으로 잇는다. `그런데 왜 하필 10초일까요? 그 값은 ...`
- 두 형태를 섞되, 질문만 쌓지 않는다. **질문 하나에는 반드시 답이 붙는다.** 답 없이 던지는 질문이 이어지면 독자는 끌려가다 지친다.

## 쓰지 않는 것

- 답이 한 줄로 끝나는 what 질문, 답 없이 분위기만 잡는 질문.
- 궁금증을 만든다고 과장하거나 낚시성으로 비트는 도입.
- 입력에 없는 동기·실패담·사람·날짜·감정, `결과적으로`·`결론적으로` 같은 결론용 추임새, 검증하지 않은 코드와 설명 없는 코드 블록.
- 불확실하면 단정하지 않는다. 추측은 추측이라고 쓰고, 확인할 것은 `확인 필요`로 남긴다.

## 자기 검토

초안을 내기 전에 확인한다.

- 도입 훅이 궁금증을 건드리고, 마무리가 그 질문에 답하는가?
- 섹션을 여는 질문이 why·what-if인가? 각 질문에 답이 붙어 있는가?
- 답의 밀도가 정의 -> 원리 -> 헷갈리는 점 -> 실제 사용 흐름을 따르는가?
- 화자를 지어내거나 대화로 꾸미지 않았는가? 글쓴이 한 명의 목소리인가?
