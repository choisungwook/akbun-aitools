# 질문식 글 골격 (annotated)

"질문이 글을 끌고 간다"는 골격이다. `{...}`는 공부한 내용으로 채우는 자리다. 화자는 글쓴이 한 명이고, 가상의 인터뷰이를 만들지 않는다.

## 1. 도입 훅 (필수)

독자가 한 번쯤 겪었거나 들어봤을 상황을 한두 문장으로 깔고, 글 전체를 관통하는 질문 하나를 던진다.

```text
{많은 사람이 공감할 상황 한두 문장}. 그런데 {그게 자주 어긋나거나 헷갈리는 지점}. 왜 그럴까요?
```

이 질문이 마무리에서 답으로 닫힌다. 훅과 마무리는 한 쌍이다. 낚시성으로 비틀지 않는다.

## 2. 질문으로 여는 섹션 (필수, 반복)

본문. 공부한 내용을 주제로 묶고, 묶음 하나를 섹션 하나로 만든다. 두 가지 형태를 섞어 쓴다.

소제목을 질문으로 두는 형태:

```text
## {why·what-if 질문 — 단정 말고 궁금증으로}

{공부한 내용으로 답한다. 정의 -> 원리 -> 헷갈리는 점 -> 실제 사용 순서로 푼다.}
**{핵심 한 문장은 굵게.}**
```

문단 첫 문장을 질문으로 여는 형태:

```text
## {주제}

{그런데 왜 하필 이렇게 동작할까요?} {바로 이어서 답한다. 질문과 답 사이를 띄우지 않는다.}
```

규칙:

- 질문은 검색하면 나오는 what이 아니라 why·what-if로 만든다.
- 한 섹션은 질문 하나로 좁힌다.
- 질문 하나에는 반드시 답이 붙는다. 답 없는 질문을 연달아 던지지 않는다.
- 질문을 붙였다고 답의 밀도를 낮추지 않는다. 질문은 입구일 뿐, 본문은 정의 -> 원리 -> 헷갈리는 점 -> 실제 사용으로 깊게 푼다.

## 3. 마무리 종합 (필수)

도입 훅의 질문으로 돌아가, 지금까지의 답을 한데 묶어 결론을 낸다.

```text
정리하면, {훅 질문에 대한 답}. {그래서 실제로 어떻게 보면 되는지 / 다음에 무엇을 보면 되는지}.
```

## 4. 참고자료 (발행용일 때 필수)

발행 가능한 글은 마지막에 `참고자료`를 둔다. 없으면 꾸미지 말고 `확인 필요`로 남긴다.

## 섹션 순서 요약

| 순서 | 섹션 | 필수 |
|---|---|---|
| 1 | 도입 훅 (상황 + 관통 질문) | 필수 |
| 2 | 질문으로 여는 섹션 (반복) | 필수 |
| 3 | 마무리 종합 (훅 질문에 답) | 필수 |
| 4 | 참고자료 | 발행용일 때 필수 |

## 짧은 예시

```text
# kubelet은 왜 lease를 10초마다 갱신할까

node가 살아 있는지 control plane은 어떻게 알까요? 답은 lease인데, 그 갱신 주기가 왜 하필 10초인지는 의외로 잘 안 알려져 있습니다.

## lease는 무엇을 대신하나

(정의) lease는 node의 생존을 알리는 가벼운 heartbeat다. ... (원리) ... (헷갈리는 점) node status update와 혼동하기 쉬운데 ...

## 그 주기를 늘리면 무엇을 잃을까

주기를 늘리면 API 부하는 줄지만, not ready 감지가 그만큼 늦어집니다. **즉 10초는 부하와 감지 지연 사이의 타협점입니다.** ...

정리하면, lease 주기는 빠른 장애 감지와 control plane 부하 사이의 균형값입니다. 이 균형을 어디에 둘지가 곧 클러스터의 장애 대응 속도를 정합니다.
```
