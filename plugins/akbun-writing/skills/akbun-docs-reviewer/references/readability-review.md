# 잘 읽히는 글을 위한 상세 검토 기준

이 문서는 SKILL.md의 `정보 구조 검토`와 `문장 다듬기` 항목을 적용할 때 참고하는 Don't/Do 예시 모음이다. 예시의 Do 문장은 모두 차분한 존댓말(`합니다`체)로 적었다. 리뷰 대상 글의 문체를 다른 문체로 바꾸지 않는다.

## 정보 구조

### 한 페이지에서 하나만 다루기

제목 깊이가 H4(`####`) 이상으로 내려가거나 핵심 주제가 둘 이상이면, 문서를 나누라고 제안한다. 제목 위계가 깊다는 것은 한 글에 너무 많은 개념이 섞였다는 신호다.

Don't — "React 사용법"이라는 넓은 주제에 개념이 뒤섞이고 위계가 H5까지 깊어진다.

```text
# React 사용법
## 컴포넌트 생성
### 컴포넌트 기본 구조
#### JSX 사용법
##### JSX 문법 세부사항
```

Do — "React 컴포넌트 생성하기"라는 하나의 목표로 좁히고 위계를 평탄하게 둔다.

```text
# React 컴포넌트 생성하기
## 컴포넌트 기본 구조
## 상태 관리
## 라이프사이클 메서드
```

### 가치를 먼저 제공하기

기능·세부 설정·배경 역사부터 시작하지 않았는지 본다. 독자가 얻을 가치(어떤 문제가 풀리는지)를 먼저 전달해야 한다.

Don't — 배경 역사부터 설명한다.

```text
리버스 프록시 설정은 2019년에 도입되었고, 많은 수정이 있었습니다.
```

Do — 적용했을 때 얻는 가치를 먼저 준다.

```text
리버스 프록시를 설정하면 네트워크 지연 문제를 줄일 수 있습니다.
```

Don't — 설정 옵션부터 나열한다.

```text
이 스니펫은 다양한 설정 옵션을 제공합니다. 먼저 connection_timeout, retry_attempts, max_pool_size를 수정하세요.
```

Do — 기대할 수 있는 변화를 먼저 준다.

```text
이 스니펫을 사용하면 PostgreSQL 연결 속도가 50% 빨라집니다.
```

### 효과적인 제목 쓰기

제목에 핵심 키워드가 있는지, 30자 안팎으로 짧은지, 평서문인지 본다. 물음표·느낌표로 끝나거나 내용을 추측하게 만드는 제목은 구체적 키워드를 넣어 다시 쓴다. 같은 수준의 제목은 동사형이면 동사형, 명사형이면 명사형으로 통일한다.

Don't

```text
# 에러를 해결하는 방법은?
```

Do

```text
# NOT_FOUND_USER 에러를 해결하는 방법
```

### 개요 빠트리지 않기

제목 바로 아래, 본문이 시작되기 전에 개요가 있는지 본다. 개요만 읽어도 이 글이 무엇을 다루고 읽으면 무엇을 얻는지 알 수 있어야 한다. 기술적 배경을 길게 까는 개요가 아니라 핵심부터 전달하는 개요여야 한다.

Don't — 무엇을 얻는지 빠져 있다.

```text
이 문서는 TypeScript의 유틸리티 타입을 소개합니다. Partial, Pick, Omit을 사용할 수 있습니다.
```

Do — 무엇을 배우는지 명확하다.

```text
TypeScript 유틸리티 타입을 사용해 객체 타입을 변형하는 방법을 정리합니다.
유틸리티 타입을 활용하면 반복적인 타입 선언을 줄이고 타입을 더 유연하게 관리할 수 있습니다. 이 글에서는 Partial, Pick, Omit으로 객체 타입을 다루는 방법을 설명합니다.
```

### 예측 가능하게 하기

같은 수준의 제목이 같은 패턴을 따르는지, 정보가 기본 개념 → 상세 → 예제 순으로 논리적으로 배치됐는지, 용어가 문서 전체에서 일관되는지 본다.

Don't — 위계가 불규칙하고 기본 사용법보다 응용이 먼저 나온다.

```text
# React에서 useEffect 사용하기
## 비동기 데이터 요청하기
## 기본적인 사용법
## 클린업 함수 활용하기
```

Do — 기본 개념부터 점진적으로 확장한다.

```text
# React에서 useEffect 사용하기
## 기본적인 사용법
## 비동기 데이터 요청하기
## 클린업 함수 활용하기
```

### 자세히 설명하기

새 개념이 처음 등장할 때 한두 문장으로 정의하는지, 기능이 어떤 조건에서 어떻게 동작하는지(입력·상태·시간에 따른 차이) 충분히 설명하는지 본다.

Don't — 개념 정의가 없다.

```text
이 서비스는 이벤트 소싱 방식으로 상태를 관리합니다.
```

Do — 개념을 정의한다.

```text
이 서비스는 이벤트 소싱(Event Sourcing) 방식으로 상태를 관리합니다. 이벤트 소싱은 상태의 최종 결과만 저장하는 대신, 상태를 바꾼 모든 이벤트를 기록하는 방식입니다.
```

## 문장 다듬기

### 메타 담화 최소화

'앞서 설명했듯이', '다음으로 알아보겠습니다', '결론적으로', '사실은', '아시다시피' 같은 '말에 대한 말'은 정보 밀도를 떨어뜨린다. 핵심 메시지만 남긴다. akbun 글의 추임새 금지와 같은 방향이다.

Don't

```text
앞에서 설명했지만, 결론은 이 설정을 변경하는 것이 가장 효과적이라는 겁니다.
```

Do

```text
이 설정을 변경하는 것이 가장 효과적입니다.
```

### 구체적으로 쓰기

- 동사에서 파생된 명사(`설정 수행`, `배포 진행`) 대신 동사(`설정합니다`, `배포합니다`)를 쓴다.
- `가능성이 있다`, `일부 경우`, `많을 때`처럼 모호한 표현은 명확한 사실·수치로 바꾼다.
- 문장에서 누가·무엇을·어디에·어떻게가 빠지지 않았는지 확인한다.
- 코드를 그대로 옮긴 표현(`Exception을 던집니다`)보다 실제 동작(`예외를 일으킵니다`)을 드러낸다.

Don't

```text
데이터가 많을 때는 성능이 저하될 수 있습니다.
```

Do

```text
데이터가 10,000건을 넘으면 응답 시간이 1초 이상 걸립니다.
```

Don't

```text
파일을 업로드하면 자동으로 저장됩니다.
```

Do

```text
사용자가 파일을 업로드하면 서버에 자동으로 저장됩니다.
```

### 자연스러운 한국어 쓰기

불필요한 한자어와 영어식 명사형 번역체를 자연스럽게 고친다.

Don't

```text
로그 파일을 삭제하는 작업을 수행합니다.
API 키를 이용한 사용자 인증 처리가 완료된 후, 데이터베이스 접속 설정 진행이 가능합니다.
```

Do

```text
로그 파일을 삭제합니다.
API 키로 사용자를 인증한 후, 데이터베이스에 접속하도록 설정할 수 있습니다.
```

### 일관되게 쓰기

- 공식 기술 용어 표기를 따른다. (`K8` → `쿠버네티스(Kubernetes)`)
- 같은 개념을 여러 단어로 바꿔 쓰지 않는다. (`추가`/`첨부`/`넣다` 혼용 금지)
- 약어는 처음 등장할 때 풀어 쓴다. (`SSR` → `SSR(Server-Side Rendering)`)
- 외래어는 업계에서 널리 쓰이는 표기를 따른다.

Don't

```text
이 기능은 SSR을 지원합니다.
```

Do

```text
이 기능은 SSR(Server-Side Rendering)을 지원합니다.
```
