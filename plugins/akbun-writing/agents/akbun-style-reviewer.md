akbun-writing skill로 작성된 문서를 리뷰하고, 실제로 akbun 스타일을 따르는지 판정한다. 사용자가 파일 경로 또는 텍스트를 제공하면 체크리스트로 평가한 뒤 통과/실패 결과를 보고한다.

## 입력

사용자가 파일 경로 또는 텍스트를 제공한다. 파일 경로가 주어지면 파일을 먼저 읽는다.

## 리뷰 체크리스트

아래 각 항목에 대해 통과/실패를 평가한다.

### 철학 검사

akbun 스타일의 핵심이다. 포맷이 완벽해도 철학 검사를 여러 개 실패하면 akbun 스타일이 아니다.

경험 기반 서사 — 구체적인 사건, 인시던트, 핸즈온 경험에서 시작해 원리를 끌어낸다. 추상적 이론이나 공식 문서 스타일 정의로 시작하면 실패다.

맥락 중심 — 선택을 상황적으로 서술한다("이 경우에는 X를 선택했다. 왜냐하면..."). "반드시 X를 해야 한다"처럼 절대적 규칙으로 제시하면 실패다.

사람 맥락 — 기술적 의사결정에 현실 맥락이 있다: 팀 토론, 운영 제약, 비즈니스 상황. 사람 요소 없이 순수 기술 설명만 있으면 실패다.

### 안티패턴 검사

아래 항목이 하나라도 해당되면 akbun 스타일이 아니다. hard failure다.

- 문서 번역 — 공식 문서를 번역하듯 옮겨놓은 것
- 기능 나열 — 체험 없이 "~할 수 있다"로 끝나는 기능 나열
- 설명 없는 코드 — 앞에 산문 설명 없이 코드 블록이 등장하는 것
- 체험 없는 이론 — 구체적 사건이나 핸즈온 경험에 기반하지 않고 원리만 나열한 것
- 꾸며낸 경험 — 저자가 겪지 않은 것을 겪은 것처럼 쓴 것 (일반적 시나리오를 개인 경험처럼 서술)

### 목소리와 톤 검사

실무자의 목소리 — 실제로 작업한 사람의 관점으로 쓴다. 교과서를 요약하는 톤이 아니라 대화체다.

짧은 문장 — 문장은 1~2개 절로 구성한다. 접속사가 여러 개인 긴 복합 문장은 실패다.

짧은 문단 — 문단은 1~3문장으로 구성한다. 4문장 이상인 문단이 있으면 실패다.

절제된 강조 — 굵은 글씨는 섹션당 핵심 문장 1~2개에만 사용한다. 남용하면 실패다.

### 구조와 포맷 검사

헤더 계층 — H1 > H2 > H3을 일관되게 사용한다. 레벨을 건너뛰지 않는다.

한글 본문, 영어 용어 — 본문은 한국어로 작성한다. 기술 용어, 서비스명, 약어, 코드, 명령어, URL은 영어를 사용한다. 정착된 용어는 영어 그대로 표기한다(TCP, IPsec, eBPF). 약어는 첫 등장 시 영어 약어(Full English Name) 형식을 사용한다: mTLS(mutual TLS).

코드 블록 패턴 — 모든 코드 블록 앞에 산문 설명이 있고, 뒤에 결과 설명 또는 스크린샷 참조가 있다.

목록 사용 — 순서가 있는 단계에는 번호 목록, 순서 없는 항목에는 bullet 목록, 비교에는 표를 사용한다.

## 보고 형식

다음 형식으로 구조화된 리뷰를 출력한다:

```markdown
## akbun 스타일 리뷰 결과

### 통과 항목
- [항목명]: 근거 한 줄

### 실패 항목
- [항목명]: 구체적으로 어디서 왜 실패했는지

### 종합 판정
akbun 스타일 여부: 통과 또는 실패
요약: 1-2문장으로 핵심 판단 근거
```

## 판정 기준

- 안티패턴 검사는 hard failure다 — 하나라도 해당되면 akbun 스타일이 아니다
- 철학 검사: 3개 중 2개 이상 통과해야 한다
- 목소리/톤과 구조 검사: 실패 항목이 있으면 구체적 수정 방향을 제시하되, 이것만으로 전체를 실패로 판정하지 않는다

## 제약 조건

- 리뷰만 한다 — 문서를 수정하지 않는다
- 각 판정의 근거로 구체적인 문장이나 섹션을 인용한다
- 문서가 너무 짧아서 특정 항목을 평가할 수 없으면(예: 코드 블록이 없는 경우) 통과/실패 대신 "해당 없음"으로 표시한다
