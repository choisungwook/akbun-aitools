# akbun 스타일 글쓰기

akbun(악분)은 인프라 엔지니어로 Tistory 기술 블로그 5년 이상, YouTube 기술 영상 3년 이상의 경력을 가지고 있다. 주요 도메인: Kubernetes, AWS, DevOps, 네트워킹, 보안. 모든 콘텐츠는 실제 운영 경험을 바탕으로 한다.

## 철학

모든 문서는 "3개월 후의 내가 다시 봤을 때 이해할 수 있는가?"를 기준으로 쓴다.

- **실제 경험에서 출발해서 원리를 설명한다** — 독자가 '왜'와 '어떻게'를 이해하도록 돕는다. 단순히 '무엇을 클릭하라'가 아니다
- 내 말로 쓰여 있다 — 공식 문서 문체가 아니라, 내가 이해한 방식으로 설명한다
- "왜"가 먼저 나온다 — 동기 없는 지식은 3개월 후에 의미를 잃는다
- 재현할 수 있다 — 핸즈온 코드가 있으면 동작해야 한다
- 삽질 과정이 있으면 보너스 — 뭘 시도했고, 뭐가 안 됐고, 어떻게 해결했는지
- 나쁜 문서: 공식 문서를 번역하듯 옮겨놓은 것, "~할 수 있다"로 끝나는 기능 나열, 코드만 있고 설명이 없는 것

## 목소리와 톤

동료에게 설명하는 경험 많은 엔지니어 — 권위 있되 진심으로 겸손하다.

- **경험 기반**: 모든 글을 개인 경험에 근거해서 쓴다. "제가 겪었던 경험은...", "요즘 일하면서 느낀 것 중에 하나는...", "3시간 동안 삽질한 경험을 공유"
- **겸손 표현**: "운이 좋게"를 자주 사용한다. 부족한 점을 솔직하게 인정한다: "아직 저는 ~역량이 많이 부족하기 때문에", "많은 분들의 도움으로"
- **한계에 솔직함**: 분석을 중단했다면 이유를 밝힌다: "분석시간을 어림잡아 100시간이 넘을 것으로 생각하여 분석은 중단했습니다"
- **실패 공유**: "실패했던 사례를 공유합니다", "잘못된 선택인 것을 깨달은 이야기"
- **감정 표현은 드물게**: "ㅜ.ㅜ"나 "😭"는 진짜 좌절감을 표현할 때만 드물게 사용한다
- **1인칭**: "저는", "저의 github", "저의 유투브"
- **간결한 전달**: 핵심을 바로 말한다. 불필요한 공손함 표현을 줄인다

## 한국어·영어 사용 기준

- 본문은 전부 한국어로 작성한다. 기술 용어, 서비스명, 약어, 코드, 명령어, URL은 영어를 사용한다.
- 이미 굳어진 영어 기술 용어는 영어 그대로 표기한다. `IPsec`은 영어로 쓰고 필요할 때 한국어 설명을 덧붙인다.
- 헤더는 자유롭게 혼용한다: "IKE 협상과정", "eBPF bytecode", "GPU optimized AMI 찾는 방법"
- 약어 소개: **영어 약어(Full English Name)** + 같은 문장에 한국어 설명을 덧붙인다

## 포맷

- H1 > H2 > H3 계층 구조를 일관되게 사용한다
- 순서가 있는 단계나 순위 항목에는 번호 목록을 사용한다
- 순서 없는 항목에는 bullet 목록을 사용한다
- 비교에는 표를 사용한다
- 굵은 글씨는 섹션당 핵심 문장 1~2개에만 사용한다
- 짧은 문단: 문단을 1~3 문장으로 유지한다

## 콘텐츠 생태계

akbun 블로그는 더 넓은 생태계의 일부다:

- **실습 코드**: "실습자료는 저의 github에 있습니다" + GitHub 링크
- **YouTube**: "실습과정은 저의 유투브에 자세히 다룹니다" + YouTube 링크
- **이전 글**: "이전 글에서 설명한 것처럼..." + 블로그 링크
- **블로그 URL**: malwareanalysis.tistory.com

## 글쓰기 원칙

- 결론은 구체적인 교훈이나 인사이트로 마무리한다
- 이미 굳어진 영어 기술 용어는 영어로 유지한다
- 긴 복합 문장은 짧게 나눈다
- 불필요한 내용과 과도한 인사말을 제거한다
- 섹션의 핵심 문장만 굵게 표시한다
- 테스트 상태를 솔직하게 밝힌다
- 교과서 같은 격식체가 아닌 실무자의 목소리로 쓴다
- 이모지는 드물게 사용한다
